# The Customization Guardrail

*Essay 5.9 — The Always-On Digital Cortex, Part 9 of 9.*

---

[Essay 5.8](05_8-historian-ratchet.html) showed three single-concern plugins composing into the historian ratchet — the ceremony that forces re-narration before plugin code edits. This final essay covers the question that gates *every* plugin-layer touch in the first place: `[PLUGIN-LOCK]`. Editing an existing plugin's code, or creating a new plugin from scratch, both start by asking that question. The question itself is gated — it admits only in two specific contexts, and the gate is what keeps the substrate intact. *[ref: plugin-lock-gates-all-plugin-work | .claude/plugins/plugin_integrity/hooks/lock-manager.sh "Gmode-OR-plugin_lock_approval gate" section | The gate fires on ALL `[PLUGIN-LOCK]` asks regardless of whether the target plugin already exists. Pre-2026-05-19 the gate guarded only existing edits (the `-d "$PLUGINS_ROOT/$target"` check). The 2026-05-19 extension dropped that guard: both editing an existing plugin AND creating a new one route through the same OR-gate (gmode OR plugin_lock_approval). The block message distinguishes the two paths in its WHY clause, but the gate logic + WHAT-TO-DO + EXPECTED-STATE are identical for both.]*

The historian ratchet asks: have you re-read the plugin's life before editing it? The customization guardrail asks something prior: are you in a context where any plugin-code work (editing OR creating) is even allowed?

For the architects in the audience. Especially the ones who plan to customize their own seed.

---

## What it owns

The customization guardrail owns the gate on **`[PLUGIN-LOCK]` admission**. Two operations route through it:

- **Editing an existing plugin's code surface** — `hooks/*.sh`, `scripts/*.sh`, `tests/*.sh`, `config.conf`, `data.json` (the last via plugin-owned scripts only).
- **Creating a new plugin** — `[PLUGIN-LOCK] <new_name>` triggers the birth branch in `lock-manager.sh`: template stamp, historian generation, baseline commit. Same gate; same admission requirement.

*[ref: customization-guardrail-owns-plugin-lock-admission | .claude/plugins/plugin_integrity/hooks/lock-manager.sh | The "Gmode-OR-plugin_lock_approval gate" sits in the PreToolUse path of lock-manager.sh: when AskUserQuestion fires the `[PLUGIN-LOCK] <target>` prefix, the hook parses the target name, checks the gate (gmode OR focused job's plugin_lock_approval), and either admits the lock for the code-surface edit or routes to the birth branch (template stamp + historian generation + baseline commit) if the target directory doesn't yet exist. As of 2026-05-19 the gate applies to BOTH paths: editing an existing plugin AND creating a new one. The free surface (CLAUDE.md, docs/*.md, voice.xml, agents/*.md) is exempted by plugin-guard.sh and edited directly by CONDENSE.]*

The gate does NOT cover a plugin's configuration / documentation surface — `CLAUDE.md`, `docs/*.md`, `voice.xml` (both `hooks/voice.xml` and `scripts/voice.xml`), and `agents/*.md` (subagent definitions, including the `historian-*.md` files). Those are free to edit without `[PLUGIN-LOCK]`. CONDENSE owns them as part of its waterfall meta-cognition step: tuning voice injections based on phase-footer notes, refreshing historian definitions, refining the working-memory CLAUDE.md. *[ref: free-surface-condense-owned | .claude/plugins/plugin_integrity/hooks/plugin-guard.sh "DOCUMENTATION + VOICE + AGENTS ALWAYS FREE" section + .claude/plugins/phase_condense/hooks/condense-guard.sh "Condense can edit .md files AND voice.xml" branch | plugin-guard.sh exempts CLAUDE.md, docs/*.md, voice.xml (any subdirectory), and agents/*.md from the PLUGIN-LOCK admission check — they pass through the lock-state filter without needing any plugin unlocked. condense-guard.sh's tool-edit branch then admits the actual write in CONDENSE. The layered design lets CONDENSE freely improve voice catalogs and historian definitions across cycles without the friction of a code-surface unlock.]*

The decision rests on one fact: editing existing plugin code (or creating a new plugin that will start running code immediately) alters the substrate form that polices every other plugin's discipline. One quiet regression deep inside a plugin's hook can poison every dispatched subagent, every phase gate, every voice fire across the session. The gate's role is to ensure plugin-code work only happens in contexts where the operator is *deliberately* doing plugin-layer work — not as a side effect of normal cycle activity. *[ref: editing-plugin-code-alters-substrate | .claude/knowledge/identity/plugin-lock-privilege.md "What does NOT need PLUGIN-LOCK" section + "The OR-gate" section | The identity deep-dive frames the qualitative difference: project edits affect the user's work; plugin code edits affect the substrate that polices every future edit, every dispatched subagent, every phase guard, every voice fire. The free surface (CLAUDE.md, docs/*.md, voice.xml, agents/*.md) doesn't share this property — those are configuration and historical-context layers that CONDENSE refines routinely. Only the code surface gets the gate.]*

## How it works — the OR-gate

The gate admits `[PLUGIN-LOCK]` on **either** of two conditions:

1. **`current_phase == "gmode"`** — the agent's focused job is currently in gmode, the operator's deliberate maintenance lane. The operator entered gmode via `[GMODE]` with a ≥100-word justification naming the work to be done. *[ref: gmode-arm-of-or-gate | .claude/plugins/plugin_integrity/hooks/lock-manager.sh "Gmode-OR-plugin_lock_approval gate" section | The gate reads `current_phase` by shelling out to `phasic_system/scripts/phase.sh current` and sets `gmode_active="true"` iff the result equals "gmode". When gmode is active, the OR-gate admits regardless of the second arm's value. The ≥100-word justification floor for `[GMODE]` is enforced separately by question-discipline + gmode-gate at question-asking time, BEFORE the agent reaches `[PLUGIN-LOCK]`.]*

2. **The focused job has `plugin_lock_approval == true`** — a top-level boolean on the job object, set when the operator answered `Approve job creation` to a `[JOB-APPROVE-CREATION]` question proposed by the agent from CONDENSE. *[ref: plugin-lock-approval-top-level-field | .claude/plugins/job_core/scripts/job.sh + .claude/knowledge/job_core/job-schema-and-extension-pattern.md "Field by field" section | New jobs are created with `plugin_lock_approval:false` as a sibling top-level field alongside `user_approval` and `depends_on` (since the 2026-05-19 schema flatten dropped the `completion_requirements` wrapper). The `--hook approve-plugin-lock` subcommand flips it to `true` on the focused job. lock-manager.sh's OR-gate reads it via `jq -r '.jobs[] | select(.focused==true) | .plugin_lock_approval // false' "$JOB_DATA"` — top-level access, no nested wrapper.]*

Either condition admits. Both failing produces a block with structured teach-text naming both admission routes. The gate's logic is short — a few lines of bash in `lock-manager.sh` — but the architectural decision behind it is deeper than the code length suggests. *[ref: or-gate-logic | .claude/plugins/plugin_integrity/hooks/lock-manager.sh "Gmode-OR-plugin_lock_approval gate" section | The gate computes `gmode_active="false"; [[ "$current_phase" == "gmode" ]] && gmode_active="true"` and reads the focused job's top-level `plugin_lock_approval` field via `jq -r '.jobs[] | select(.focused==true) | .plugin_lock_approval // false' "$JOB_DATA"`. Block fires iff both arms fail. Block voice: `plugin-lock-requires-gmode-or-approved-job` — names both admission paths in its teach-text and clarifies that the same gate applies to both editing and birth. Tests at test-lock-manager.sh Groups 24-25 verify both arms across editing AND birth scenarios.]*

## Why both routes, not one

Each route covers a case the other can't reach without contortion.

**gmode** is for ad-hoc work the operator initiates. The operator notices a problem with a plugin and decides to fix it right now. There's no pre-existing job for this work; making one would be overhead for what may be a one-line fix. The operator enters gmode (a `[GMODE]` question with a ≥100-word justification), does the work, exits. The trigger is the operator's noticing. *[ref: gmode-operator-initiated-quick-fixes | .claude/knowledge/identity/plugin-lock-privilege.md "What each context is for" section + "Why both, not one" section | The identity deep-dive frames gmode as "for quick, ad-hoc operator-initiated work that fits in a few actions: fix a deadlock, repair a misconfigured plugin, patch a regression spotted mid-cycle." Entry cost: the ≥100-word justification + the operator's awareness that they're outside cycle discipline. gmode bypasses ALL phase guards (sentinel locks, point budgets, altered-list scoping) so the agent has the broadest write surface — appropriate for quick repair work that doesn't fit normal OPEVC cycles.]*

**User-approved jobs** are for planned work the agent initiates. Any phase (OBSERVE / PLAN / EXECUTE / VERIFY) can notice that a customization to the plugin layer would help — a pattern that should harden into a plugin, a voice that keeps misfiring, a workflow the always-on layer doesn't yet support. *[ref: any-phase-notices | .claude/knowledge/identity/user-approved-jobs.md "The proposal flow" Step 1 | The identity deep-dive frames Step 1 as "any phase (OBSERVE / PLAN / EXECUTE / VERIFY) can recognize that a customization to the plugin layer would help" — the noticing is recorded in the phase's own CLAUDE.md footer as a [PENDING-JOB] marker or free-form rationale for CONDENSE to consume. The split between noticing (any phase) and proposing (CONDENSE-only) keeps job-creation decisions in the cycle-wide synthesis phase.]*

The noticing phase drops a note in its CLAUDE.md footer; CONDENSE's waterfall step 3 consumes the note and fires `[JOB-APPROVE-CREATION] <name>` from CONDENSE. The user confirms; the new job is created with `plugin_lock_approval=true`; `[PLUGIN-LOCK]` admits inside that job for the duration of the customization work. The trigger is the agent's noticing; the creation is CONDENSE's responsibility. *[ref: condense-only-creates-the-job | root CLAUDE.md "CONDENSE.consume-markers sub-operation" + .claude/knowledge/identity/user-approved-jobs.md "The proposal flow" section + .claude/plugins/job_core/hooks/question-capture.sh "[JOB-APPROVE-CREATION] phase-of-firing gate" section | The root CLAUDE.md frames CONDENSE.consume-markers as step 3 of the waterfall — it greps each phase footer for `[PENDING-JOB]` markers and dispatches creation. The phase-of-firing gate in question-capture.sh blocks `[JOB-APPROVE-CREATION]` outside CONDENSE with the `job-approve-creation-condense-only` voice teaching the agent to drop a note in the footer instead — CONDENSE has cycle-wide context for what follow-up work is needed. The gmode-bypass that applies to most phase guards does NOT extend to job creation; per user directive "condense is the only phase that can create these jobs by asking the job creation questions."]*

The two routes match the two natural triggers (operator-noticed vs agent-noticed) without forcing every plugin-layer touch through the same ceremony.

## The propose-and-confirm flow

The `[JOB-APPROVE-CREATION]` route is where most customization happens, because it's where the agent's noticing meets the operator's judgment. The flow:

1. **Any phase notices a need.** During OBSERVE / PLAN / EXECUTE / VERIFY, the agent identifies that a customization to the plugin layer would help — a pattern that should harden, a voice that keeps misfiring, a workflow the always-on layer doesn't yet support. The noticing phase drops a note in its CLAUDE.md footer (a `[PENDING-JOB]` marker or free-form rationale). *[ref: any-phase-notices-records-in-footer | .claude/knowledge/identity/user-approved-jobs.md "The proposal flow" Step 1 + .claude/plugins/phase_condense/CLAUDE.md "consume-markers" waterfall step | Any working phase can notice a need; the noticing is recorded in the phase's own CLAUDE.md footer for CONDENSE to consume. CONDENSE's waterfall step 3 (consume-markers) reads the footer notes and decides whether to surface them as `[JOB-APPROVE-CREATION]` proposals. The split (any phase notices, only CONDENSE proposes) keeps job-creation decisions in the cycle-wide synthesis phase where the agent has full context.]*

2. **CONDENSE proposes via `[JOB-APPROVE-CREATION] <name>`.** Job creation is CONDENSE's responsibility. The question body names the proposed objective, the plugins that would be edited (or created), the rationale (which note from earlier phases surfaced the need), and the expected scope (single-cycle DEEP, multi-cycle, approximate effort). The phase-of-firing gate in `question-capture.sh` blocks the prefix outside CONDENSE — if any other phase tries to fire it, the agent gets a teach-voice pointing back to the footer-note channel. *[ref: condense-only-proposal-via-gate | .claude/plugins/job_core/hooks/question-capture.sh "[JOB-APPROVE-CREATION] phase-of-firing gate" section + .claude/plugins/job_core/hooks/voice.xml `job-approve-creation-condense-only` block | The Pre handler in question-capture.sh checks `current_phase != "condense"` against the `[JOB-APPROVE-CREATION]` prefix and exits 2 with the block voice. The voice teaches the agent the WHY (CONDENSE has cycle-wide context), the WHAT-TO-DO (drop a [PENDING-JOB] note in the current phase footer), and the EXPECTED-STATE on retry (current_phase == condense). The gate covers ALL non-CONDENSE phases including gmode — per user directive, "condense is the only phase that can create these jobs."]*

3. **The user judges.** Standard AskUserQuestion options: `Approve job creation` confirms; `Redirect` lets the user adjust scope before approval; `Reject` drops the proposal. The user holds the architectural decision; the agent holds the operational details. *[ref: standard-approve-job-creation-options | .claude/plugins/job_core/hooks/question-capture-hook.sh "JOB-APPROVE-CREATION Handler" section | The Post handler matches the literal answer string `Approve job creation`; any other answer (including `Redirect` or `Reject`) falls through to the not-granted voice. The literal string is the contract between the AskUserQuestion option list (offered by the agent at proposal time) and the handler that materializes the approved job. Changing the option label requires changing this grep-string in lockstep.]*

4. **On approval, the Post handler creates the new job and flips its `plugin_lock_approval` flag.** *[ref: post-handler-create-then-approve | .claude/plugins/job_core/hooks/question-capture-hook.sh "JOB-APPROVE-CREATION Handler" section | The handler extracts the proposed name from the question text, calls `bash "$JOB_SH" create "$new_name"` to capture the new job id, then `bash "$JOB_SH" --hook approve-plugin-lock "$new_id"` to flip the flag. The new job lives in the standard data.json structure, distinguishable from typical jobs only by the top-level `plugin_lock_approval == true`. Voice `job-approve-creation-result` confirms with `new_id` + `new_name` substituted and reminds the agent to focus + activate the new job when ready.]*

5. **The agent focuses and activates the new job** when ready to begin substrate work. The pending job sits in the data.json until the agent (or operator) decides to switch contexts. Inside the activated approved job, `[PLUGIN-LOCK]` admits without further questioning — the gate sees `plugin_lock_approval=true` on the focused job and admits the lock for both editing existing plugins AND creating new ones. *[ref: pending-job-then-focus-activate | .claude/plugins/job_core/scripts/job.sh focus + activate subcommands + .claude/plugins/plugin_integrity/hooks/lock-manager.sh OR-gate | The newly approved job is created `status:"pending", focused:false` with top-level `plugin_lock_approval=true` already set (via the `--hook approve-plugin-lock` call inside the Post handler). The agent later runs `job.sh activate <new_id>` (flips status to active; clears `plugin_lock_approval=false` if reactivating a completed job — stale approvals don't leak through reactivation) then `job.sh focus <new_id>` (clears focus on all other jobs; sets focused=true on the new one). Once focused, lock-manager.sh reads the top-level `plugin_lock_approval` field — the second arm of the OR-gate — and admits PLUGIN-LOCK inside the job for both editing existing plugins and creating new ones.]*

## What would break without it

Without the customization guardrail, plugin code edits and births would be admitted in any cycle, in any phase, by any job. That isn't paranoia — it's the failure mode of every agent design where the agent can rewrite the rules that constrain it without ceremony. A buggy hook can poison every subsequent tool call. A regressed test guard can let broken plugins through. A misnamed voice id can silently fail to coach the agent through a critical moment. A spurious new plugin can hijack hook events the operator didn't authorize. The substrate's integrity is what makes the rest of the always-on layer trustworthy; without the gate, that integrity dissolves whenever the agent is mid-cycle with broad write access. *[ref: substrate-corruption-failure-modes | .claude/knowledge/identity/plugin-lock-privilege.md "The OR-gate" section | The identity file names the corruption surface directly: "Plugin code edits affect the substrate that polices every future edit, every dispatched subagent, every phase guard, every voice fire." That makes plugin-layer changes qualitatively different from project file changes — a regression deep in a plugin's hook can corrupt every downstream decision the agent makes for the rest of the session. The gate exists because the cost of admitting a bad edit isn't local; it's substrate-wide.]*

Worse: without a structured gate, customization decisions get made ad-hoc, in the middle of work, under deadline pressure — exactly the situation where careful architectural judgment is hardest. The gate forces the customization decision into one of two deliberate moments: the operator entering gmode with a justification, or the user judging a `[JOB-APPROVE-CREATION]` proposal CONDENSE surfaced. Both moments slow the agent down, on purpose. Slowness is the point. *[ref: two-deliberate-moments-friction-by-design | .claude/knowledge/identity/plugin-lock-privilege.md "Why both, not one" section | The identity file frames the two contexts as covering the two natural triggers (operator-noticed vs agent-noticed-then-user-judged) without forcing every plugin-layer touch through the same ceremony. The friction (≥100-word gmode justification OR user-approval question + proposal body) is the architectural feature, not a bug — both moments interrupt the work-pace deliberately so the customization decision gets fresh judgment rather than running on autopilot.]*

## What you would customize

The customization guardrail is the rare plugin-layer surface where the design assumes most architects will *inherit, not rewrite*. The two-route architecture is the architectural decision; the specific prefix names and voice wording are the surface knobs.

You would tune the **proposal threshold**. The current prototype waits for the agent's own noticing — a pattern recurs, a voice keeps misfiring. Your seed may want a more aggressive proposer (every CONDENSE, list candidate plugin-improvements; reduce to one and propose) or a more conservative one (only propose after the user has expressed friction). The thresholds live in plugin code; the architectural fact (proposals come from agent noticing + user judging, with CONDENSE doing the asking) doesn't move. *[ref: proposal-threshold-customization-knob | .claude/knowledge/identity/user-approved-jobs.md "The proposal flow" Step 1 | The identity file frames the noticing-to-proposal pipeline: any phase notices, drops a note, CONDENSE consumes via waterfall step 3, CONDENSE proposes via `[JOB-APPROVE-CREATION]`. The specific noticing thresholds (how aggressive, which signals count as "the agent noticed") are not codified as named parameters in the current prototype; the architectural fact (any-phase notices, CONDENSE proposes, user judges) is what doesn't move. A customizer's seed can encode its own threshold by adding noticing logic to a phasic hook (OBSERVE through VERIFY) and a corresponding marker-consumption rule in CONDENSE's waterfall.]*

You would rename the **`[JOB-APPROVE-CREATION]` prefix** if your operator vocabulary uses different words. The prefix is registered in `question_discipline`'s PREFIX_REGISTRY as a single literal; the Post handler in `job_core` greps for the same string; the phase-of-firing gate in `question-capture.sh` matches the same literal. A consulting seed might prefer `[ENGAGEMENT-AUTHORIZED]`; a research seed might use `[INVESTIGATION-PROTOCOL]`. The mechanism is the architecture; the words are yours. *[ref: prefix-registry-rename-touch-points | .claude/plugins/question_discipline/hooks/question-discipline-gate.sh PREFIX_REGISTRY + .claude/plugins/job_core/hooks/question-capture.sh "[JOB-APPROVE-CREATION] phase-of-firing gate" + .claude/plugins/job_core/hooks/question-capture-hook.sh "JOB-APPROVE-CREATION Handler" | The `PREFIX_REGISTRY` bash array in `question-discipline-gate.sh` holds the registered prefix. Renaming `[JOB-APPROVE-CREATION]` is a one-line edit to that array (the registration side). The Pre handler in `question-capture.sh` (phase-of-firing gate) and the Post handler in `question-capture-hook.sh` (creation handler) both grep for the same literal — three sites touched in lockstep for a rename. The comment block above the registry names the three steps explicitly.]*

You would rewrite the **gate's block voice** to match your operator's reading level. The current voice has a 3-part teach: WHY (substrate edits matter) + WHAT TO DO (the two routes) + EXPECTED STATE. Your seed may want a shorter voice for an architect-level operator who already knows the why, or a longer one for a new operator still learning the surface. The voice lives in `voice.xml` — and editing `voice.xml` doesn't require `[PLUGIN-LOCK]` itself, because CONDENSE owns voice tuning as a routine meta-cognition step. *[ref: gate-block-voice-3-part-structure | .claude/plugins/plugin_integrity/hooks/voice.xml `plugin-lock-requires-gmode-or-approved-job` block | The block voice follows a literal three-part shape (WHY / WHAT TO DO / EXPECTED STATE). WHY explains that any plugin-layer change (edit OR birth) alters the substrate. WHAT TO DO instructs the agent to pick one of `[GMODE]` (≥100-word justification) or `[JOB-APPROVE-CREATION] <name>` (objective + plugins + rationale, fired from CONDENSE). EXPECTED STATE names the retry precondition (`current_phase == gmode` OR focused job's `plugin_lock_approval == true`). Rewriting means editing one `<block>` element in voice.xml — free without PLUGIN-LOCK.]*

You would extend the **approval-flag granularity**. The current design has one top-level flag (`plugin_lock_approval`). Your seed may want finer granularity — a flag per concern, or per plugin family. The schema is flat and extensible: adding a flag is a schema extension + a new `--hook approve-X` command. *[ref: approval-flag-granularity-extension-pattern | .claude/plugins/job_core/scripts/job.sh `--hook approve-plugin-lock` subcommand + .claude/knowledge/job_core/job-schema-and-extension-pattern.md "Field by field" section | New jobs are created with top-level `user_approval:false, plugin_lock_approval:false, depends_on:[]` (since the 2026-05-19 schema flatten dropped the `completion_requirements` wrapper). The `approve-plugin-lock` subcommand is hook-gated and flips `plugin_lock_approval=true` via update_job. Adding a new approval concern (e.g., `secrets_approval` for a `secrets_guard` plugin) means three edits: extend the create-job literal to include the new field, add a parallel `--hook approve-secrets` subcommand, and wire a Post handler in the owning plugin to flip it on the user's "Approve" answer.]*

What you would NOT do is remove the gate. The gate is what makes the architecture safely modifiable. A seed that admits PLUGIN-LOCK in any context is a seed that will eventually corrupt its own substrate, usually under deadline pressure when the architect's judgment is weakest. *[ref: gate-removal-breaks-substrate-safety | .claude/knowledge/identity/plugin-lock-privilege.md "Implementation status" section + frontmatter description | The identity file's frontmatter description names the gate's role directly: "Editing or creating any plugin alters the substrate that enforces every other plugin's discipline." Removing the OR-gate while keeping plugin-code edits possible would let any cycle modify the substrate without operator awareness — the exact failure mode the architecture exists to prevent. The gate is the load-bearing piece that turns a code-rewriting agent into one whose self-modification is safely operator-gated; replace its admission logic with anything weaker and you lose the property the rest of the system depends on.]*

## What the gate teaches

The customization guardrail is the architectural conclusion of the prior essays in this series. The always-on plugin layer holds the agent's reflexes; the substrate holds its working memory; the historian ratchet forces re-narration before edits; the gate forces deliberate context before any plugin-code edit (or new-plugin birth) happens at all. Each layer adds a discipline; the gate is the one that turns the system into something safely modifiable *by you* — not by the original developer, by you, the operator who installed the seed last week or last month or last year.

That is the [agent-developer-user → agent-user collapse](../b8/08_1-apprentice-to-architect-foundation.html) made operational. The agent proposes customization. You judge it. The substrate enforces the discipline that keeps your judgment safe. No developer in the loop. And the canonical reference the seed itself consults when reasoning about its own design is this nine-essay series — plus [Essay 6](../b6/06_1-phasic-foundation.html), [7](../b7/07_1-plugin-kit-foundation.html), [8](../b8/08_1-apprentice-to-architect-foundation.html) and the `.claude/knowledge/` topic files you cultivate as you customize. The PowerPoint of seed agents lives here.
