# The Plan File — Stages and Completion

*Essay 6.10 — The Markov Phasic Brain, Part 12 of 13.*

---

[Essay 6.9](06_9-gmode.html) opened the off-cycle lane — the deliberate maintenance side-quest where the phase gates are dropped so the operator can do work that doesn't fit the standard OPEVC shape. Gmode covers a *short* horizon: hours, one sitting, one ceremony. The opposite horizon — work that doesn't fit inside a single cycle and shouldn't — needs a different mechanism: a document on disk that remembers the job across days and dozens of cycles.

This essay maps the skeleton of that mechanism: what the plan file owns, how a job's Stage is decided, how a multi-cycle job knows it is done, and where the file lives. The next essay turns that skeleton into long-horizon memory.

---

## What the plan file owns

The plan file owns the long-horizon memory of a job. One field carries it — `plan_file`, living on the job's `phase_plan` extension (keyed to the job by its ID, stored in `phase_plan`'s own `data.json`): the literal `false` when the job has no plan, or the name of a working document on disk (`<name>.md` or `<name>.yaml`) when it does. The document is the memory; the field is the pointer. *[ref: plan-file-field-three-values | .claude/context/job-stages-plans.md "Job Stage" section | The canonical glossary defines `plan_file` as a single value per job — one of `false` (no plan), `<name>.md`, or `<name>.yaml`. A once-only guard prevents flipping the extension on the same job. The job's Stage is read directly from this shape: `false` is Stage 1, `.md` is Stage 2, `.yaml` is Stage 3.]*

The plan file is born in cycle 1 EXECUTE — not at job creation. Every job is created the same minimal way: a name and a short objective. The `plan_file` pointer starts out *undecided* — a set-once sentinel that has not yet been resolved to a value. Nothing on disk yet. The decision of whether this job *needs* a long-horizon document — and what shape it takes — is made later, once the agent has actually looked at the work. *[ref: plan-file-born-cycle-1-execute | .claude/context/job-stages-plans.md "Plan file lifecycle" section | The glossary is explicit: the on-disk plan file is BORN in cycle 1 EXECUTE, not at job creation. At creation the `plan_file` pointer is the undecided set-once sentinel (null), carried on the job's phase_plan extension — the job record itself has no plan_file key. The cycle-1 PLAN of the activated job owns the decision and calls `set-plan-file` (`false` for Stage 1, `<name>.md` for Stage 2, `<name>.yaml` for Stage 3); the advance out of PLAN is blocked until that call resolves the sentinel. EXECUTE is the phase that creates non-CLAUDE.md files, so it is the phase that writes the Stage 2/3 plan document to disk.]*

What lives where matters. Job-scoped facts live on the job object; file-scoped facts live in the file. Plan files carry a small identification frontmatter — a `job:` line and a `plan_file:` line declaring which job they belong to — plus a `total_cycles:` count (for `.md`) or a `cycles:` list (for `.yaml`) declaring how many cycles the work is planned to take. There is no lifecycle-state field; the job's progress is read directly from its `cycle` counter against that declared total. *[ref: plan-frontmatter-and-cycle-count | .claude/context/job-stages-plans.md "Job Stage" section | Stage 2 plans carry "required YAML frontmatter declaring `total_cycles: <N>`"; Stage 3 plans declare a `cycles:` list whose `max(c.cycle for c in cycles)` is the total. The job's completion eligibility is computed from `current_cycle` against that total (see "Cycle counter behavior"), not tracked by a separate state field.]*

## The Stages

A job's *Stage* is its classification by plan-file shape, and it is decided in **cycle 1 PLAN** — not at creation. The cleanest mental model: cycle 1 of any multi-cycle job is for creating or updating its plan; cycle 2 onward is for the actual work. Fresh job? Cycle 1 creates the plan. Re-activated job whose plan already exists? Cycle 1 updates it. Real work always starts cycle 2. *[ref: stage-decided-cycle-1-plan | .claude/context/job-stages-plans.md "Job Stage" section | The glossary: Stage is decided in cycle 1 PLAN of the activated job, NOT at creation. The creation objective only IMPLIES a possible Stage; the real decision is made once OBSERVE has gathered the job's context. In cycle 1 PLAN the seed makes the 3-way call via `set-plan-file` — `false` (Stage 1), `<name>.md` (Stage 2), or `<name>.yaml` (Stage 3) — and the PLAN→EXECUTE advance is blocked until that call lands, so the Stage decision cannot be skipped (even Stage 1 requires an explicit `set-plan-file false`). The Stage is then read from `plan_file`'s decided shape; the decision is owned by the activated job's first PLAN phase, not by whatever created it.]*

**Stage 1 — single-cycle, collaborative.** Cycle 1 PLAN makes the explicit Stage-1 call — `set-plan-file false` — resolving the sentinel to `false`. There is no document on disk. The entire OPEVC pass holds the work in working memory and commits the artifacts directly. Single fixes, single updates, a freestyle teaching conversation — anything that closes in one cycle. Backward edges happen *inside* the cycle, between phases, not across cycles. A Stage-1 job can also be the seed from which a repeatable Stage-2 version grows: at CONDENSE, the agent can emit a `[PENDING-JOB]` marker proposing the multi-cycle version of the same work. *[ref: stage-1-no-plan-file | .claude/context/job-stages-plans.md "Job Stage" section | Stage 1 is defined as single-cycle, collaborative. Cycle 1 PLAN calls `set-plan-file false`, resolving the undecided sentinel to the decided value `false` — even Stage 1 requires the explicit call, since the PLAN→EXECUTE advance is blocked while the pointer is still undecided. ONE cycle that loops internally via backward transitions; ONE CONDENSE at the end where CONDENSE asks `[JOB-COMPLETE]`. Used for one-off teaching, exploration, simple fixes. At runtime, `plan_file == false` is the decided-state signal the eligibility gate reads as Stage 1.]*

**Stage 2 — repeatable, with an `.md` plan.** Cycle 1 PLAN calls `set-plan-file <name>.md`; cycle 1 EXECUTE creates the plan file at the job's own run-aware directory, `.claude/jobs/<job_id>/plan.md`, with `total_cycles: <N>` in its frontmatter. The document becomes the long-horizon memory of this job: cycles 2..N do operational work against it, VERIFY refines it when a cycle surfaces a better criterion, and every cycle's PLAN re-reads it at phase entry. This is what carries a long job's understanding across days, after chat history has long since rolled off. *[ref: stage-2-md-plan | .claude/context/job-stages-plans.md "Job Stage" section | Stage 2: "Cycle 1 PLAN calls `set-plan-file <id> <name>.md`. Cycle 1 EXECUTE creates the plan file at the job-level `.claude/jobs/<job_id>/plan.md` (the run-aware job directory; pre-relocation plans sat under `knowledge/plans/`) with required YAML frontmatter declaring `total_cycles: <N>`. … Cycles 2..N do operational work against the plan. … Reactivatable — the plan file persists on disk; subsequent activations get smarter plans."]*

**Stage 3 — repeatable, with a `.yaml` plan.** Cycle 1 PLAN calls `set-plan-file <name>.yaml`; cycle 1 EXECUTE writes a structured document whose `cycles:` list declares each cycle's expected work. In completion semantics it is **identical to Stage 2** — cycle-1 plan creation, cycle-2..N work, the same counting rule deciding when the job is done. The only difference is the file format, and what that format unlocks: the `.yaml` can inject itself into the agent's context at every phase entry. Like any Stage, Stage 3 is a valid starting format — a job can be born Stage 3, decided in cycle-1 PLAN. In practice the `.yaml` is usually authored once a mature Stage-2 `.md` pattern has run enough times to be worth hardening into structure — a fresh job inspired by that pattern, never an automatic flip of the Stage-2 job itself. *[ref: stage-3-identical-except-format | .claude/context/job-stages-plans.md "Job Stage" section | Stage 3 is defined as "Otherwise IDENTICAL to Stage 2 in completion semantics … The ONLY difference between Stage 2 and Stage 3 is plan-file format. Inspired typically by a mature Stage-2 job repeated many times — NOT a state-flip on the Stage-2 itself." This corrects the older model that framed the `.yaml` as a separate dependent job spawned from the `.md` job.]*

A future *Stage 4* — a job that ships its own plugin, adding new gates and new voices — is named in the architecture as a future direction, not part of the current prototype. *[ref: stage-4-aspirational | .claude/context/job-stages-plans.md "Job Stage" section | "A future Stage 4 is named in B8.2 as the 'plugin form of a job' — a job that ships its own per-job plugin adding new hard gates and new voices. No code referent yet (aspirational per Rule 38's customization-gradient Layer 5)."]*

## How a multi-cycle job knows it is done

There is no state machine walking a plan through named stages. There is a counting rule. *[ref: no-state-machine-only-a-counting-rule | .claude/context/job-stages-plans.md "Plan file lifecycle" + .claude/context/job-completion-reactivation.md "Cycle counter behavior" sections | The lifecycle has no separate state field and no named approval stages — "Plan file lifecycle" tracks the plan with the `plan_file` value alone (`false`, an `.md`, or a `.yaml`). What governs a multi-cycle job's progress is the cycle counter against the plan's declared total: "Cycle counter behavior" defines `cycle` as monotonic-increment-only, and completion is the counting rule `current_cycle >= effective_last_cycle`, not a walk through named stages.]*

Each multi-cycle plan declares its total — `total_cycles` for an `.md`, the length of the `cycles:` list for a `.yaml`. The job carries a `cycle` counter. The job becomes *eligible* to complete when its current cycle reaches the **effective last cycle**: the declared total, plus any extension cycles the run had to add when the original plan came up short. *[ref: effective-last-cycle-formula | .claude/context/job-completion-reactivation.md "Cycle counter behavior" section | The glossary gives the formula `effective_last_cycle = (plan_file != false ? plan_file.total_cycles : 1) + extension_cycles_added`. The `[JOB-COMPLETE]` eligibility check at CONDENSE fires for Stage 2/3 when `current_cycle >= effective_last_cycle`. The `extension_cycles_added` term lets a run exceed its planned length when work surfaces mid-run (see "Extension cycle").]*

That second term is a small per-run counter on the job — `extension_cycles_added`, starting at zero and climbing by one each time a cycle has to be added. The extension is a *safety valve*, not a normal route: it catches whole-job work that slipped past the cycle's own checks and only surfaced at the end. It is what lets the planned total be an estimate the run can honestly exceed rather than a hard ceiling: a `.md` plan that declared four cycles but needed a fifth carries `extension_cycles_added = 1`, and the eligibility check moves with it. It is a *per-run* counter — it resets to zero whenever the job re-activates for a fresh run, so each run forecasts against its own plan rather than inheriting the last run's overruns. *[ref: extension-cycles-added-per-run-counter | .claude/context/job-completion-reactivation.md "`extension_cycles_added` field" + "Cycle counter behavior" sections | The glossary defines `extension_cycles_added` as a per-run integer counter starting at 0, incremented by each `--add-cycle "<text>"` invocation, feeding `effective_last_cycle = (plan_file != false ? plan_file.total_cycles : 1) + extension_cycles_added`. "Cycle counter behavior" lists it among the fields reset to 0 on reactivation (alongside `cycle` and `user_approval`); it does NOT reset on a plain `activate`. The field, the `--add-cycle` command, and the formula are built: `phase_plan/scripts/plan.sh` creates `extension_cycles_added: 0` at job birth, increments it (alongside an `extension_contexts[]` append) on each `--add-cycle`, and resets it to 0 on reactivation; the eligibility gate in `job_core/hooks/question-capture.sh` reads `current_cycle >= effective_last_cycle`.]*

For a Stage-1 job there is no plan file, so `plan_file == false` makes it eligible to complete at *every* CONDENSE — every cycle is potentially the last. For Stage 2 and Stage 3, eligibility waits until the counter reaches the planned end. The value of `plan_file` is doing double duty: it names the document, and its presence-or-absence is what tells the completion gate which rule to apply. *[ref: completion-eligibility-by-plan-file | .claude/context/job-completion-reactivation.md "[JOB-COMPLETE] completion hooks" section | The pre-completion hook's eligibility gate: "`plan_file == false` (Stage 1, fires every CONDENSE) OR `current_cycle >= effective_last_cycle` (Stage 2/3)." A single field, `plan_file`, both points at the document and discriminates Stage 1 from Stage 2/3 at the gate.]*

When the final cycle's CONDENSE arrives and the job is eligible, the agent asks the operator one structured question — `[JOB-COMPLETE]` — presenting the evidence that the objective is met. (That ceremony lands inside CONDENSE, the cognitive organ from [Essay 6.7](06_7-condense.html).) Approval flips the job to `completed`. The `plan_file` is not nulled or archived; it stays exactly where it is. *[ref: completion-plan-file-persists | .claude/context/job-stages-plans.md "Plan file lifecycle" section | On approval: "`job.sh approve` flips `user_approval = true` → post-completion hook calls `job.sh complete` → status flips to `completed`. `plan_file` STAYS at its value across the completion." The plan is neither nulled nor moved on completion — it persists exactly so it can be re-read on the next activation.]*

That persistence is the whole point of the learning loop the next essay turns to.

VERIFY guards each cycle along the way. If a cycle's `## Outstanding Items` are not empty, VERIFY blocks the forward-advance and the agent loops back to PLAN or EXECUTE to resolve them. And in cycle 1 — the plan's one editing window — if VERIFY *edits* the plan file (refines a vague criterion, drops one that can't be tested) it cannot forward-advance either; it backward-advances to PLAN. From cycle 2 onward the plan is frozen for the run: an operational cycle routes a plan problem through its outstanding items or an extension cycle, never by editing the plan. PLAN re-cognizes and routes one of two ways: backward to OBSERVE when the refined criteria need more context, or straight forward to VERIFY to re-test the updated plan without re-executing. Post-creation plan-file edits belong to VERIFY alone, inside that cycle-1 window — when PLAN wants the plan changed, its change-requests ride the working CLAUDE.md, and VERIFY applies them. The loop ends when a VERIFY pass makes no edit and the cycle advances. *[ref: plan-verify-backward-loop | .claude/context/opevc-phases.md "Plan-verify backward loop" section | Triggered when VERIFY edits the plan file: the custom VERIFY gate detects the edit and BLOCKS forward-advance; the agent calls `verify-commit.sh --backward plan`. VERIFY is the plan file's only post-creation editor — PLAN never writes it; PLAN's change-requests ride the CLAUDE.md layer and PLAN routes either backward to OBSERVE or forward to VERIFY directly (the plan→verify forward transition). Convergence: a VERIFY pass that makes NO edit forward-advances to CONDENSE; typically 1-3 iterations for cycle-1 plan establishment.]*

## Where the plan file lives

The plan file's home is the job's own **run-aware directory**, `.claude/jobs/<job-id>/` — *run-aware* because each activation of the job, reactivations included, gets its own numbered `run-N/` space — where everything a job accumulates across its runs lives together, each run a recorded practice the future seed can mine. Plans written before this relocation still sit under the legacy `.claude/knowledge/plans/` path, which every consumer reads as a fallback; new plans land in the job directory. *[ref: run-aware-job-directory | .claude/context/job-system.md "Run-aware job directory" section | The consolidated term defines `.claude/jobs/<job_id>/` as the per-job directory that homes everything a job accumulates across its runs: a job-level `plan.{md,yaml}` plus `run-<r>/` subdirectories holding `compaction-<seq>.md` (one per session in the run) and `session-log-<c>.md` (CONDENSE output, one or more per run). It folds in two relocations — the plan file moves from `.claude/knowledge/plans/` to job level, and the CONDENSE step-7 session archive moves from `.claude/knowledge/session/` to `run-<r>/session-log-<c>.md`. The plan-file relocation is built — every consumer resolves the job-level home first, `knowledge/plans/` honored only for pre-existing plans; the session-archive relocation is also built — `session-log.sh` writes the run-aware `run-<r>/session-log-<c>.md` and `condense-commit.sh`'s archive-verified gate blocks deflation until it is captured, with pre-relocation archives still readable at the legacy `knowledge/session/` path.]*

The layout puts the plan at the **job level** and everything per-run beneath it:

```
.claude/jobs/<job-id>/
  plan.md | plan.yaml        # job-level: persists and refines across runs
  run-1/
    compaction-1.md          # one per session in the run
    session-log-1.md         # CONDENSE output; one or more per run
  run-2/
    ...
```

The plan file sits at the job level, not inside any run — so a reactivation reads the same refined plan the last run left behind, which is the learning loop the next essay maps. What lives *inside* each `run-<r>/` is that run's record: the **compaction files** the agent seals at each session boundary, and the **session logs** CONDENSE writes at the end of a cycle. Over many runs the seed can read across `run-*/` to learn how it does this kind of job — which is the point of recording each run as practice.

The plan file shares this directory with a sibling document that points the opposite way. The plan file is **forward and operational** — it carries cross-*cycle* work state: what each cycle does, its acceptance criteria, the structure the next cycle inherits.

The **compaction file** from [the always-on cortex](../b5/05_3-brain-guard.html) is **backward and reflective** — it carries cross-*session* cognitive state: how the work went, the loops the agent got stuck in, the blind spots it hit, and the handful of top-level facts that must survive the next context clear. *[ref: compaction-vs-plan-boundary | .claude/context/job-stages-plans.md "Plan file lifecycle" section (Boundary vs the compaction file) + .claude/context/brain-memory.md "Compaction file" section | The Plan-file-lifecycle boundary paragraph states the plan file carries cross-cycle WORK state and is forward/operational, while the compaction file carries cross-session COGNITIVE state and is backward/reflective; both now live in the run-aware job directory — the plan file at job level (relocated from `.claude/knowledge/plans/`), the compaction files under `run-<r>/`. The Compaction file term (.claude/context/brain-memory.md) defines it as the per-job cognition-about-the-job, built across phase exits and carried across sessions, never condensed. pending: code not yet aligned.]*

The two never blur. Ask of any note: is this what the next *cycle* needs to do, or what the next *session* needs to remember? Work state goes in the plan; cognition about the work goes in the compaction file. They are siblings in the same job directory precisely because keeping them apart — one forward, one backward — is what lets a long job both *run* coherently across cycles and *think about itself* across sessions.

This run-aware home is where the plan file, the compaction files, and the session logs all live — the job's own directory holds everything a run accumulates, with pre-relocation archives still readable at the legacy `knowledge/` paths.

The plan file's structure, its Stages, the counting rule that closes it, and its on-disk home are the *skeleton* of long-horizon memory. What turns that skeleton into a living memory — the `.yaml` that injects itself at every phase entry, the loop that lets a repeating job get smarter run-over-run, and the bridge to building new cognition — is the final essay of the series.
