# The Plan-State Machine — Long-Horizon Memory

*Essay 6.10 — The Markov Phasic Brain, Part 10 of 10.*

---

[Essay 6.9](06_9-gmode.html) opened the off-cycle lane — the deliberate maintenance side-quest where the phase gates are dropped so the operator can do work that doesn't fit the standard OPEVC shape. Gmode covers a *short* horizon: hours, one sitting, one ceremony. This final sub-essay covers the opposite horizon — work that doesn't fit inside a single cycle and shouldn't. Work that needs to remember itself across days and dozens of cycles.

The mechanism is small. Fields on the job object, a small state machine, named approval gates. The discipline it produces is what turns the markov brain from a single-session cognition into something that can hold a plan in its head for weeks. *[ref: plan-state-machine-substrate | .claude/plugins/phase_plan/scripts/plan.sh:334-906 | The entire plan-state machine is implemented across the public command `set-plan-file` (L334-384) and the hook-only commands `approve-md` (L849-862), `approve-yaml` (L867-878), and `seal-plan` (L884-894). Each handler reads `plan_state` from `data.json`, dies if the current state doesn't match the expected source state, and writes a single jq update flipping the field. The state is never inferred from the file — it is the authoritative job-scoped fact, and the handlers are the only legal mutation paths.]*

This sub-essay is for you — especially when your work takes longer than one sitting and the plan needs to outlive any single cycle.

---

## What it owns

The plan-state machine owns the long-horizon memory of a job. The fields on the job object inside `data.json` — `plan_file` and `plan_state` — carry the state: `plan_file` names the working `.md` document on disk (or holds the literal `false` when the job has no plan), and `plan_state` tracks where the document is in its lifecycle. *[ref: plan-fields-on-job-object | .claude/plugins/job_core/scripts/job.sh:177 | job_core's create handler initializes every new job with the literal shape `plan_state:"none"` alongside `completion_requirements`, `user_interactions`, and the other job-level fields. The plan_file field is added later by `plan.sh set-plan-file` in PLAN of cycle 1; until then the job carries no plan. Both fields live on the job object — never in plan-file frontmatter — so the lifecycle state spans the .md and .yaml documents as one job-scoped fact.]*

The `plan_state` field cycles through the stages of a plan's life: `drafting` (a working `.md` plan is accumulating refinement across cycles), `md_approved` (the `.md` has passed user judgment and the next cycle is dedicated to creating a `.yaml`), `yaml_drafting` (a parseable `.yaml` companion is being authored alongside the `.md` and may already feed draft phase-entry context), `yaml_ready` (the `.yaml` has passed approval and now injects itself as trusted context into every subsequent cycle's phase entry), and `sealed` (the optional terminal state where the pair is archived and no longer injects). A default value, `none`, covers jobs that never get a plan file at all. *[ref: plan-state-named-stages | .claude/plugins/phase_plan/scripts/plan.sh:647 | The plan-state command comment lists the canonical state set the prototype ships with: `none | drafting | md_approved | yaml_drafting | yaml_ready | sealed`. New jobs initialize with `plan_state:"none"` (job.sh:177 create handler). The extension-based dispatch in set-plan-file (L358-380) sets `drafting` on `.md`, `yaml_drafting` on `.yaml`. The approve-md / approve-yaml / seal-plan handlers flip to the remaining stages. The set is the prototype's current shape; the "What you would customize" section discusses extending it.]*

What lives where matters. Job-scoped facts live on the job object; file-scoped facts live in the file. Plan files themselves carry a small identification frontmatter — a `job:` line and a `plan_file:` line — declaring which job they belong to, and nothing more. The lifecycle state is intentionally not in the file. It is in `data.json` because the state describes the *whole plan lifecycle* that spans both documents; the files are the artifacts, the state describes the pair. *[ref: plan-file-frontmatter-identification-only | CLAUDE.md "Job Forms (OBSERVE cycle 1 classifies)" section | Root brain spells out the storage contract: "`plan_file` and `plan_state` live on the job object in `data.json` (managed via `plan.sh set-plan-file` / `approve-md` / `approve-yaml` / `seal-plan`). Plan files themselves carry identification frontmatter only (`job:` + `plan_file:` at top); the lifecycle state is on the job, not in the file." The split exists so the state describes the pair of documents as one job-scoped fact rather than living redundantly in two files that could diverge.]*

## The three job forms

The state machine produces a small set of canonical job forms, each fitting a different shape of work. OBSERVE in cycle 1 classifies the job into one of them. The classification decides whether a plan file will ever exist, and which lifecycle the plan will follow. The cleanest mental model: **cycle 1 of any multi-cycle job is for creating or updating its plan file; cycle 2 onward is for the actual operational work.** Fresh job? Cycle 1 creates. Re-activated job whose plan file already exists? Cycle 1 updates. Real work always starts cycle 2. *[ref: three-job-forms-canonical | CLAUDE.md "Job Forms (OBSERVE cycle 1 classifies)" section | Root brain canonically names the three forms: (1) Single-cycle deep, plan_file=false, freestyle and collaborative, backward edges and loops inside one cycle; (2) Multi-cycle with .md plan job, plan_file=<name>.md and plan_state walks drafting → md_approved; (3) Multi-cycle with .yaml plan job — a SEPARATE dependent job spawned from CONDENSE of a completed Form-2 md-job. The dep yaml-job carries plan_file=<same_basename>.yaml and plan_state walks yaml_drafting → yaml_ready → (optional) sealed. The .md and .yaml share basename and sit side-by-side in `.claude/knowledge/plans/`, but they live on two distinct jobs with two distinct lifecycles. Form 3 is not an evolution of Form 2's job; it is Form 2's child.]*

**Form 1 — single-cycle deep.** The job sets `plan_file = false`. There is no document on disk; the entire OPEVC pass holds the work in working memory and commits the artifacts directly. Single bug fixes, single feature adds, single doc updates, freestyle teaching conversations — anything that closes in one cycle. Backward edges happen *inside* the cycle, between phases, not across cycles. The job completes and the memory of it lives only in git history and the `knowledge/` files CONDENSE produces. A single-cycle job can also be the *seed experience* from which Form-2 and Form-3 jobs grow: at CONDENSE, the agent emits a `[PENDING-JOB]` marker proposing the repeatable version of the same work as a Form-2 md-plan job. *[ref: form-1-no-plan-file-false-branch | .claude/plugins/phase_plan/scripts/plan.sh:351-355 | The `false` branch of set-plan-file writes a single jq update — `(.plan_file = false | .plan_state = "none")` — and prints "PLAN FILE: none (false). No plan file required for this job." The false branch is symmetric with the .md and .yaml branches but explicit: the job is opting out of long-horizon memory. The `none` state has no entry hook and no injection; the cycle is the only memory.]*

**Form 2 — multi-cycle md-plan job.** The job sets `plan_file = <descriptive_name>.md` and `plan_state = drafting`. The document becomes the long-horizon memory of THIS job: PLAN names it in cycle 1, EXECUTE creates the first draft (or updates an existing one on re-activation), VERIFY refines each cycle to record where the work has reached. From cycle 2 onward, PLAN re-reads it at phase entry so the cycle inherits the contract earlier cycles wrote. This is what carries a long job's understanding across days when chat history has long since rolled off. The job's lifecycle ends at `md_approved` — at which point its work is done; what follows is a separate job, not a continuation of this one. *[ref: form-2-md-extension-drafting | .claude/plugins/phase_plan/scripts/plan.sh:358-384 | The extension-based dispatch makes Form 2 implicit in the filename: any `.md` value to set-plan-file routes to `new_state="drafting"` and the jq write becomes `(.plan_file = $v | .plan_state = "drafting")`. The handler then prints "PLAN FILE (md-job): … execute phase will create the first draft in knowledge/plans/. plan_state=drafting." The agent declares the form by naming the file extension; the state machine takes over from there. set-plan-file's L364-365 guard rejects a second call if plan_file is already set — meaning the same job can never hold both a .md plan and a .yaml plan; the .yaml lives on a separate dep job.]*

**Form 3 — multi-cycle yaml-plan job (dep of a Form-2 md-job).** Form 3 is its own job, not a continuation of Form 2. The Form-2 md-job's CONDENSE — once `plan_state = md_approved` and the operator has approved the `.md` — emits a `[PENDING-JOB]` marker for a dependent yaml-job whose `plan_file = <same_basename>.yaml`. When the dep yaml-job is activated and focused, ITS cycle 1 PLAN calls `set-plan-file plan_<same_basename>.yaml`, the extension parser sets `plan_state = yaml_drafting` directly (skipping `drafting` / `md_approved` — those belong to the parent job), and ITS cycle 1 EXECUTE creates the `.yaml`. From cycle 2 onward, the dep job does real operational work shaped by the `.yaml` injecting per-phase context at every phase entry. The .md (parent's plan_file) and the .yaml (this job's plan_file) coexist on disk, same basename, side-by-side — coupled by naming convention, not by job lineage. *[ref: form-3-is-dep-yaml-job-not-state-flip | .claude/plugins/phase_plan/scripts/plan.sh:364-365 set-plan-file once-only guard + L697-701 NOTE | The set-plan-file handler dies if plan_file is already set: "Plan file already set to: $current_pf. Cannot change." This forces .md and .yaml to live on different jobs — the same job cannot transition its plan_file from .md to .yaml. The L697-701 NOTE makes the dependent-jobs flow explicit: "a dep yaml-job calls 'set-plan-file <id> plan_<name>.yaml', and the extension-based plan_state logic ... auto-sets plan_state = 'yaml_drafting' on the dep yaml-job. md-jobs end at md_approved and never flip to yaml_drafting." The dep yaml-job is created via CONDENSE's standard pending-job pattern; its plan_state lifecycle is wholly independent of the md-job's.]*

## The two state machines

The plan-state machine is actually *two* state machines, one per job. The Form-2 md-job has its own lifecycle; the Form-3 dep yaml-job has its own. They are coupled by the dep-job link and the shared basename of their plan files, but their states are independent. Each transition is deterministic, owned by exactly one phase, and triggered by one specific event.

```
md-job  (Form 2):  none → drafting → drafting (loop) → md_approved  [job ends — its planning work is done]

[handoff: CONDENSE of the md-job emits [PENDING-JOB] for the dep yaml-job, same basename]

dep yaml-job (Form 3):  none → yaml_drafting → yaml_drafting (loop) → yaml_ready → (optional) sealed
```

### md-job transitions

**`none → drafting` (md-job, cycle 1).** PLAN of the md-job's cycle 1 calls `plan.sh set-plan-file <name>.md`. The extension drives the state: `.md` lands in `drafting`. EXECUTE creates the first draft of the file during the same cycle (on re-activation when the file already exists, EXECUTE links to it without overwriting and VERIFY updates from there). *[ref: none-to-drafting-md-extension-trigger | .claude/plugins/phase_plan/scripts/plan.sh:368-384 | The extension parser switches on `${local_val##*.}`: `.md` sets `new_state="drafting"`, `.yaml` sets `new_state="yaml_drafting"`. The jq write at L379 — `(.plan_file = $v | .plan_state = $s)` — does both fields atomically so the agent cannot land in a state where one is set without the other. The handler also prints the file_kind ("md-job" / "yaml-job") so the agent sees what form just got declared. The L395-399 branch handles file-already-exists ("linking to existing plan") vs first-creation ("execute phase will create the first draft").]*

**`drafting → drafting` (md-job, cycle 2+).** Subsequent cycles refine the `.md`. VERIFY edits it — appending status rows, captured lessons, completion marks — and can backward to PLAN or EXECUTE when re-analysis is needed. PLAN reads it at phase entry. The loop can run for as many cycles as the work needs. Many jobs never leave this state; the `.md` is enough. *[ref: drafting-loop-verify-edits-required | .claude/plugins/phase_verify/scripts/verify-commit.sh:167-177 | The verify-commit gate enforces the loop discipline: when `plan_file` is active, VERIFY must show at least one edit to the plan file before forward-commit is allowed (`plan_edits ≥ 1`). The `plan_file` value is read via `bash "$PLAN_SH" --hook plan-file`. The "no edits" path dies with a teach-text directing the agent to update the plan with where the work has reached. The state stays `drafting` indefinitely until `approve-md` fires.]*

**`drafting → md_approved` (md-job, VERIFY).** When satisfied with the plan, VERIFY asks the user via `[PLAN-APPROVAL]` with an "Approve .md plan" option. On approval, VERIFY calls `plan.sh approve-md`, which flips `plan_state` from `drafting` to `md_approved`. The md-job's planning lifecycle ends here — its job is to have a good prose plan, and that goal is met. *[ref: approve-md-state-flip | .claude/plugins/phase_plan/scripts/plan.sh approve-md command | The approve-md handler reads the current plan_state, dies if it is not "drafting", and writes a single jq update flipping the field to "md_approved". No archive, no file motion — approval is a state flip, not a retirement. The handler is reserved for hook mode (HOOK_MODE=true) so only the question-capture flow can invoke it; the agent cannot self-approve. The handler's success message names the next step: "Next: at CONDENSE of this md-job, emit a [PENDING-JOB] marker for a dependent yaml-job (same basename, .yaml extension). The dep yaml-job's cycle 1 PLAN calls set-plan-file plan_<name>.yaml; cycle 1 EXECUTE creates the .yaml; the md-job ends here."]*

### The handoff: md-job's CONDENSE spawns the dep yaml-job

Once `plan_state = md_approved`, the md-job has reached the end of its planning lifecycle. The next step is **a separate job, not a continued state transition**. At any CONDENSE phase after approval (typically the same cycle that landed the approval), the agent emits a `[PENDING-JOB]` marker proposing a dependent yaml-job: same basename, `.yaml` extension, the same operational objective, this time encoded as per-phase voice augmentations rather than prose. The operator reviews the pending job and activates it when ready. Once focused, that dep job runs its OWN OPEVC cycles — and the yaml state machine below begins. *[ref: condense-spawns-dep-yaml-job-as-pending | .claude/plugins/phase_condense/agents/condense-job-creator.md + the [PENDING-JOB] marker pattern | CONDENSE step 3 of the 7-step waterfall consumes `[PENDING-JOB]` markers from the cycle's phase footers; the condense-job-creator subagent calls `job.sh create-dependent <parent_id> <name>` with an objective describing the dep yaml-job's purpose. The parent is the md-job; the dependent is the yaml-job. The dependent inherits the parent's basename by convention, not by code — the agent names it intentionally (`plan_<same_basename>.yaml`) so the side-by-side coupling reads naturally on disk. Until the operator activates the pending dep yaml-job, the md-job's `md_approved` state is its terminal — it can complete or stay paused with no further cycles.]*

### dep yaml-job transitions

**`none → yaml_drafting` (dep yaml-job, cycle 1).** PLAN of the dep yaml-job's cycle 1 calls `plan.sh set-plan-file plan_<basename>.yaml`. The extension parser sees `.yaml` and sets `plan_state = yaml_drafting` directly (the dep job never visits `drafting` or `md_approved` — those belong to its parent). EXECUTE of the same cycle creates the `.yaml` on disk; VERIFY refines (cycle-1 verify is allowed to push back to EXECUTE or PLAN until the .yaml is good — same write-once-then-edit-in-verify pattern the md plan uses). On re-activation, the .yaml already exists; cycle 1 becomes the update cycle. *[ref: dep-yaml-job-cycle-1-creates-yaml | .claude/plugins/phase_execute/hooks/execute-guard.sh yaml branch + .claude/plugins/phase_plan/scripts/plan.sh:386-392 yaml extension parser | The execute-guard's yaml-write branch requires plan_state=yaml_drafting AND plan_file ending in .yaml — the canonical signal that the focused job IS the dep yaml-job. The legacy same-job yaml_required precondition was retired (2026-05-23) once the dep-jobs model became canonical; plan_state + plan_file extension are sufficient. The .yaml is created Write-only on a non-existent path; subsequent edits route through VERIFY of the same dep job.]*

**`yaml_drafting → yaml_drafting` (dep yaml-job, cycle 2+).** Real operational work cycles. The .yaml's `cycles[N].phases.<phase>.<voice-id>` keys inject per-phase voice augmentations at every phase entry; the agent doesn't re-read the plan, the plan re-reads itself into the agent's voice stream. VERIFY can still edit the .yaml when a cycle surfaces a refinement; backward to PLAN is allowed when schema-level re-analysis is needed. The loop runs until the .yaml captures every phase's contribution well enough to graduate. *[ref: yaml-drafting-loop-verify-edits-required | .claude/plugins/phase_verify/scripts/verify-commit.sh:167-177 | The same plan_edits gate covers the dep yaml-job's yaml_drafting loop — VERIFY must show at least one edit before forward-commit. Backward to PLAN is the legal path when the schema needs re-analysis. The loop runs cycles until `approve-yaml` fires; the gate keeps the agent from forwarding the cycle without contributing to the yaml.]*

**`yaml_drafting → yaml_ready` (dep yaml-job, VERIFY).** VERIFY asks the user via `[YAML-APPROVAL]` with an "Approve .yaml plan" option. The validator gate forces the question into the right shape: only the VERIFY phase may ask, the question must carry the plan file name in its text, both an approve option and a Review option must be present, and the question must run at least the configured word floor (currently 100 in the prototype, tunable per seed via `COMPLETION_QUESTION_WORD_MIN`). On approval, VERIFY calls `plan.sh approve-yaml`, flipping the state. The .yaml's injection now carries the operator's trust stamp — the same injection mechanism that ran in `yaml_drafting`, just with the "this is the right shape" gate cleared. *[ref: yaml-approval-validator-gate | .claude/plugins/job_core/hooks/question-capture.sh:137-203 | The validator branches on the prefix kind (`[PLAN-APPROVAL]` expects plan_state=drafting and approve label "Approve .md plan"; `[YAML-APPROVAL]` expects plan_state=yaml_drafting and approve label "Approve .yaml plan"). It then enforces five conditions: A7 phase-of-firing gate (block unless current_phase == verify), plan_file must be set, plan_state must match the expected stage, the question must include the plan_basename or its .yaml companion, and word count ≥ COMPLETION_QUESTION_WORD_MIN (default 100). Each failure exits with a distinct voice id (plan-approval-wrong-phase, plan-approval-no-file, plan-approval-state-mismatch, plan-approval-filename-required, plan-approval-word-count) for clean teach-text.]*

**`yaml_ready → sealed` (dep yaml-job, optional terminal).** When the long-horizon work is truly done, VERIFY may call `plan.sh seal-plan`. This is the only command that archives: this dep job's `plan_file` (the `.yaml`) is pushed onto a `completed_plan[]` list on this job, `plan_file` flips back to `false`, and `plan_state` lands on `sealed`. Approval was a state flip; sealing is a retirement. The distinction matters — many `yaml_ready` dep jobs run for cycles after approval, with the `.yaml` injecting its context each time. Sealing closes that loop on purpose. *[ref: seal-plan-terminal-archive | .claude/plugins/phase_plan/scripts/plan.sh:881-894 | The seal-plan handler is explicit about its terminal role. The comment names it "Terminal: archives plan_file to completed_plan[], sets plan_state=sealed, plan_file=false." The handler dies unless current state is `yaml_ready` ("Approval and the yaml stage must complete before sealing."). The jq update is one expression: `(.completed_plan += [$pf] | .plan_file = false | .plan_state = "sealed")` — archiving and state flip happen atomically. After sealing, the `.yaml` no longer injects; the dep yaml-job has retired its long-horizon memory. The parent md-job's `.md` is untouched (it lives on the parent's plan_file or in its own completed_plan[] if separately sealed).]*

## How `.yaml` injection turns the brain long-horizon

The `.md` is for humans. The `.yaml` is for the brain.

Once a job has an active YAML companion — `plan_state` is `yaml_drafting` or `yaml_ready`, and `plan_file` still points at the plan — the phasic system's entry hook reads the `.yaml` on every phase transition and writes the phase-specific field-map to a cache file. The shared voice helper reads that cache when it fires any voice during the new phase, and yaml field-map entries augment the relevant voices — turning a job's structured long-horizon context into a stream of injections that show up automatically wherever the agent is reading guidance. *[ref: phase-init-yaml-injection-on-transition | .claude/plugins/phasic_system/hooks/phase-init.sh write_yaml_cache + phase-transition branch | The write_yaml_cache helper resolves `$ROOT_DIR/.claude/.tmp/yaml-injection-cache.json` and writes the field-map for the focused job's current phase by calling `bash "$PHASE_SH" --hook read-yaml`. The phase-transition branch of phase-init.sh calls write_yaml_cache after every advance, back, or commit-transition event so the freshly entered phase's yaml content immediately augments downstream voices. Gmode enter clears the cache (side-quest isolation); gmode exit restores it. New jobs and focus events also refresh. Net result: phase entry is the injection moment; the .yaml plays back the job's long-horizon memory each time the agent crosses a phase boundary.]*

The augmentation has three modes. A bare-string yaml entry **appends** its text to the rendered voice after a blank line — the back-compat default. A structured entry of the form `{mode: replace, text: "..."}` **replaces** the rendered voice text entirely with the yaml content (useful when the job's context-specific voice should override the universal one). A `{mode: prepend, text: "..."}` entry **prepends** the yaml content before the rendered voice (useful for stamping a job-specific banner above the standard guidance). All three forms apply the same variable substitution as the underlying voice — `{{job_id}}`, `{{focused_name}}`, and the rest substitute identically into the yaml text. *[ref: voice-helper-three-mode-augmentation | .claude/plugins/lib/voice-helper/voice-helper.sh yaml-cognition block | The helper inspects the cache entry's JSON type per voice id. Bare string → APPEND (back-compat); object `{mode, text}` → switch on `mode` between `append` / `replace` / `prepend`; unknown mode falls back to APPEND so unrecognized values cannot silently drop a voice. The same KV_ARGS used to interpolate the xml voice text are re-applied to the yaml text via a preserved-args pattern so substitution keys stay symmetric across the two surfaces.]*

The yaml key IS the voice id directly — no transform, no naming convention. That makes the **callable voice catalog** the contract surface for layer-4 customization: a yaml plan can only target voices that some hook or script actually fires. A new helper builds the catalog by scanning every plugin's `voice.xml` for `id="..."` definitions and every non-test `.sh` / `.py` for quoted references. A defined voice with no callsite is an **orphan** — useless, because it never injects into context, and an invisible trap for yaml authors who target it and watch nothing happen. The seed agent ships with a recurring cleanup job pattern that consumes the orphan list and either links or removes each one over time. *[ref: voice-catalog-and-orphan-discipline | .claude/plugins/lib/yaml-cognition/build-voice-catalog.py + CLAUDE.md | The catalog tool enumerates defined ids across all `voice.xml` files plus quoted references in non-test plugin source (recognizing direct `get_voice "id"`, the `block "id"` wrapper, case-dispatch branches, and array enumerations like `COACHING_IDS=("id" ...)`). Modes: `--audit` (full report), `--list-callable` / `--list-orphans` / `--list-missing` (one id per line). The lib's CLAUDE.md names the contract: a yaml plan's keys must satisfy the callable set; orphans are scheduled for cleanup via a recurring yaml job, not allowed to linger as silent traps.]*

The yaml plan loader enforces the contract at validation time. When PLAN calls `validate-yaml-format` on a new or edited `.yaml`, the schema check is followed by a key-vs-catalog check: every key under `cycles[*].phases.<phase>` must match a callable voice id, otherwise the plan is rejected with a `did you mean` suggestion drawn from the closest matches. This catches the most common yaml authoring error — typing a voice id that no longer exists, or never did — before the job ever runs the cycle. *[ref: yaml-key-validator-rejects-uncallable-ids | .claude/plugins/lib/yaml-cognition/validate-yaml-voice-ids.py wired in .claude/plugins/phase_plan/scripts/plan.sh validate-yaml-format | The validator loads the callable set from the catalog builder, walks `cycles[*].phases.<phase>.<key>` (both current and legacy schema shapes), and rejects keys outside the callable set with `difflib.get_close_matches` suggestions. plan.sh's `validate-yaml-format` handler calls the validator AFTER the schema check passes; rejection fires the `error-yaml-voice-ids-invalid` voice with the path + flattened detail. Exit 0 = all keys callable; exit 1 = one or more invalid (report to stderr); exit 2 = catalog build error.]*

YAML jobs also skip the manual forecast ceremony each phase normally requires. When a job's `plan_file` ends in `.yaml`, every phase's init stamps the new entry's multiplier with `3` (the RAPID end of the scale) instead of the `0` sentinel that would otherwise leave tools locked until the agent calls `set-multiplier`. The forecast lives in the yaml plan itself; the agent shouldn't be asked to re-forecast at every phase entry. The PRE-SET sentinel voice — the one that nags the agent to set a multiplier — naturally goes silent for these jobs because it only fires when the multiplier is `0`. *[ref: yaml-jobs-auto-multiplier-3 | .claude/plugins/job_core/scripts/job.sh default-multiplier + each phase's render_X_entry default_mult | A new hook-only command on job_core, `default-multiplier`, delegates to PLAN_SH and returns `3` when the focused job's `plan_file` ends in `.yaml`/`.yml`, else `0`. Each phase script (observe/plan/execute/verify/condense) calls this from init via a `resolve_default_multiplier` helper and passes the value through `render_X_entry()` → `create_X_job_with_entry()` / `append_X_entry()` as a positional default. The jq update in render overrides the template's hardcoded `multiplier: 0` with the resolved value. For YAML jobs whose plan_file is set BEFORE first OBSERVE init, all five phases auto-multiplier-3 from cycle 1; for jobs classified during cycle 1, OBSERVE+PLAN still go through forecast (correct — that's the classification window), and from `set-plan-file <name>.yaml` onward every subsequent init auto-multiplier-3. The sentinel-PRE-SET voice in observe-sensor.sh only fires when `current_mult == 0`, so YAML jobs never see it.]*

This is the long-horizon mechanism. Without it, every cycle would start cold — the agent would have to re-read the `.md` and re-derive what each phase should know. With it, the agent's context grows back the long-form structure of the job automatically, each cycle, at every phase boundary. The brain doesn't have to remember to remember. *[ref: phase-entry-is-injection-moment | .claude/plugins/phasic_system/hooks/phase-init.sh:130-141 | The phase-transition branch calls write_yaml_cache after every advance, back, and commit-transition event. The write resolves the focused job's current phase and calls `bash "$PHASE_SH" --hook read-yaml` to extract the field-map for that phase, writing it to `$ROOT_DIR/.claude/.tmp/yaml-injection-cache.json`. Subsequent voices read the cache automatically. Without this hook, the cache wouldn't refresh on phase boundaries and the `.yaml` would be dead documentation; the hook IS the injection discipline.]*

The `.md` and `.yaml` are not redundant — they are different surfaces. The `.md` is a prose document a human can scan to follow the arc of a long job. The `.yaml` is a parseable structure the brain can inject at exactly the right moment. They coexist; neither replaces the other. *[ref: md-yaml-coexist-different-surfaces | .claude/plugins/phase_plan/scripts/plan.sh:516-586 | The validate-yaml-format command checks the `.yaml` against the phase.sh read-yaml v2 Path C schema — phases as top-level keys, each phase containing structured fields the injection target understands. The validator's existence is the point: the `.yaml` is *for* a parser, not a human. The `.md` carries no such validator and is never parsed; it is for the partner reading the long-horizon arc. The two files share a basename so the agent knows they belong together, but they answer different needs.]*

## What would break without it

Take the state machine out. Now every multi-cycle job has to hold its own plan in working memory or re-derive it each session. The seed agent can still attempt long-horizon work, but each cycle starts from scratch — re-reading whatever notes happen to survive in `knowledge/`, re-building intuition that the previous cycle already had. Long jobs drift. A late cycle of a multi-cycle migration contradicts an early one because it never read the earlier plan. *[ref: state-machine-substrate-is-multi-cycle-memory | .claude/plugins/phase_plan/scripts/plan.sh:647 | The plan-state command exists precisely so other phases can ask "what stage is this job's plan at?" without re-parsing the file. Remove the field and the question has no authoritative answer; remove the handlers (set-plan-file, approve-md, approve-yaml, seal-plan) and every cycle has to reconstruct the lifecycle from heuristics. The state machine IS the substrate that lets phases coordinate across cycles without rebuilding the contract every time.]*

Take the approval gates out. Now the `.md` and the `.yaml` exist but the operator never gets to say "this plan is right." The agent can rewrite the plan freely each cycle; the long-horizon "memory" becomes whatever the agent last decided to write. The user's architectural judgment — the thing that makes the seed safely modifiable by the operator instead of by the developer — is removed from the long-horizon loop. *[ref: hook-mode-prevents-agent-self-approval | .claude/plugins/phase_plan/scripts/plan.sh:850,868,885 | The approve-md, approve-yaml, and seal-plan handlers all open with the same guard: `[[ "$HOOK_MODE" != "true" ]] && die "Command '…' is reserved for system automation."`. Only the question-capture hook flow — which fires when the operator clicks "Approve" in an AskUserQuestion — sets HOOK_MODE=true and runs the flip. The agent cannot invoke these commands directly. Remove the validator (question-capture.sh:137-203) and HOOK_MODE never gets set legitimately; remove the HOOK_MODE guard and the agent can self-approve, which is the failure the gate exists to prevent.]*

Take the file-based companions out (the `.md`-only world). Now the long-horizon context is a prose document the agent has to remember to re-read every cycle. Sometimes it does. Sometimes it doesn't. Injection at phase entry is the discipline that turns "should re-read the plan" into "the plan re-reads itself into the context, every phase, every cycle." *[ref: yaml-companion-is-injection-target | .claude/plugins/phasic_system/hooks/phase-init.sh:55-65 | The write_yaml_cache helper reads `${YAML_INJECTION_CACHE:-$ROOT_DIR/.claude/.tmp/yaml-injection-cache.json}` and writes the focused job's current-phase field-map. The cache only exists because the `.yaml` exists. Without a parseable companion, there is nothing the cache can hold; the voice helper has nothing to append; the agent gets generic voices instead of context-loaded ones. The `.yaml` is not redundant prose — it is the injection target, and without it the long-horizon-context-into-voices mechanism has no source.]*

Each layer of the machine — the state field, the approval gates, the parseable injection target — closes one failure mode of long-horizon work that ordinary chat-history-based agents fail at.

## What you would customize

The plan-state machine is the rare mechanism where the design assumes most architects will *extend it, not replace it*. The state field is extensible, the prefixes are renamable, the `.yaml` schema is yours to design.

You would extend the **`plan_state` values themselves**. The prototype ships a handful of active states (the set is named in `plan.sh`'s plan-state command and stays small on purpose). Your seed may want intermediate stages — a `peer-review` state between `drafting` and `md_approved` for jobs that pass through a second human; a `paused` state for long jobs the operator wants to suspend without sealing; an `archived-but-active` state for plans that keep injecting `.yaml` even after the work is done because the long-horizon context is reusable. Adding a state is a schema extension plus a handler command on `plan.sh`. *[ref: extending-plan-state-is-additive | .claude/plugins/phase_plan/scripts/plan.sh:846-906 | Each existing transition is one handler block on plan.sh: a HOOK_MODE guard, a current-state read, a die-if-wrong-state check, and a single jq update flipping the field. Adding a new state means adding one handler in the same shape (and registering the source/target states it transitions between). The data layer is just a string field on the job object; the only schema discipline is keeping the handler set consistent so phases that read plan-state get expected values back.]*

You would rename the **approval prefixes** to match your operator vocabulary. The current `[PLAN-APPROVAL]` and `[YAML-APPROVAL]` are descriptive but generic; a legal-domain seed might prefer `[ENGAGEMENT-APPROVED]` and `[MEMO-LOCKED]`, a research seed might use `[PROTOCOL-APPROVED]` and `[DATASET-LOCKED]`. The prefixes are registered in one place; the validator branches on the prefix kind to set expected states and approve labels; renaming is a one-line change plus a voice-text update. *[ref: prefix-renaming-is-registry-edit | .claude/plugins/lib/prefix-registry.conf + .claude/plugins/job_core/hooks/question-capture.sh:137-203 | The prefix is named in the registry config (where `[PLAN-APPROVAL]` and `[YAML-APPROVAL]` are listed alongside `[PLUGIN-LOCK]`, `[TEST-LOCK]`, `[GMODE]`, `[JOB-COMPLETE]`, `[POINT-BOOST]`, `[WAITING]`, `[PENDING-JOB]`, `[JOB-APPROVE-CREATION]`). The validator branches on `prefix_kind`; the expected `plan_state` and approve-label are set per branch. Renaming a prefix means changing the string in the registry and the branch label; the gate machinery does not care about the name, only the shape.]*

You would design the **`.yaml` schema**. What fields does each phase need at entry? OBSERVE may want a `recall_targets:` list; PLAN may want a `decisions_locked:` map; EXECUTE may want a `forbidden_files:` array; VERIFY may want a `acceptance_criteria:` list with pass/fail rows. The current prototype ships a minimal injection — one objective per phase. Your seed will discover what richer fields produce richer cognition for your work, and the `.yaml` is where that discovery lives. *[ref: yaml-schema-is-customizable | .claude/plugins/phase_plan/scripts/plan.sh:516-586 | The validate-yaml-format handler checks the `.yaml` against the phase.sh read-yaml v2 Path C contract — phases as top-level keys, structured fields under each. The schema is enforced by the validator, but the validator is yours to edit: add expected fields, change validation rules, define new injection paths. The minimal version the prototype ships is a starting point, not a ceiling. The brain's long-horizon cognition is exactly as rich as the schema you author.]*

You would tune the **seal-plan timing**. The current rule is "seal after work is truly done, and stop injecting." Your seed may want automatic sealing when a completion flag flips, operator-only sealing through a ceremony question, or a separate never-seal / archived-active state that keeps injecting the `.yaml` forever as durable institutional memory. The mechanism is one command; the policy is yours. *[ref: seal-policy-is-yours | .claude/plugins/phase_plan/scripts/plan.sh:881-894 | The seal-plan handler implements the *mechanism* — die-if-not-yaml_ready, push plan_file onto completed_plan[], flip plan_file=false and plan_state=sealed. The *policy* — when to call it — is not codified anywhere; the prototype leaves it as a manual VERIFY decision. Wrap seal-plan in a job_archiver hook that fires on completion_requirements pattern matches, gate it behind a [SEAL-PLAN] question, or remove the call entirely so plans inject forever. The handler doesn't constrain when it runs.]*

You would adjust the **multi-cycle threshold**. The current decision rule lives in OBSERVE cycle 1's narrative judgment: "big project + likely review-and-revision cycles → form 2." That is operator-judgment-shaped; if your seed runs in a domain with a clear length threshold (say, every job past a certain phase-count threshold goes multi-cycle), you can codify it as a hint voice or a hard gate in OBSERVE. *[ref: cycle-1-classification-is-judgment-shaped | CLAUDE.md "Job Forms (OBSERVE cycle 1 classifies)" section | Root brain lists the decision rules as narrative heuristics, not code: "Generic conversational request, no expected repeat → form 1. Repeatable structured work that benefits from accumulating plan → form 2. Big project work that spans multiple phases + likely review-and-revision cycles → form 2." OBSERVE in cycle 1 applies them as judgment. Codifying a hard gate means adding a voice or a script that fires on a measurable trigger (phase count, prior-job-similarity, domain tag) and steers the classification. The prototype keeps it judgment-shaped because the trade-offs are different per seed.]*

What you would NOT do is remove the machine. Take it out and the brain reverts to single-session cognition. The whole markov phasic discipline — the cycle, the cognitive metabolism organ, the multiplier, gmode — is built on the assumption that some jobs deserve to be remembered across cycles. The plan-state machine is what makes that remembering deterministic instead of accidental.

Lift the pattern out of the seed agent and into a law practice. A partner drafting an engagement letter could carry the `.md` → `.yaml` lifecycle the same way: the prose draft accumulates with the partner across weekly client-review cycles in `engagement.md`, an `[ENGAGEMENT-APPROVED]` answer flips `plan_state` from `drafting` to `md_approved`, the next cycle derives `engagement.yaml` with parseable fields the agent injects at every subsequent contract-revision phase — scope clauses, fee structure, retention terms, conflict-check results — so each follow-up cycle inherits the engagement's structure automatically instead of re-reading the prose. *[ref: pattern-lifts-to-law-practice-engagement-letter | CLAUDE.md "Job Forms (OBSERVE cycle 1 classifies)" section | The form-3 lifecycle the root brain names — `.md` drafting → `[PLAN-APPROVAL]` → `.yaml` drafting → `[YAML-APPROVAL]` → injection at every phase entry — is generic enough that any long-horizon professional artifact (an engagement letter, a research protocol, a treatment plan, a project charter) fits the same shape. The prefixes rename, the schema customizes, the lifecycle stays. That generality is why the prototype puts the lifecycle in `plan.sh` and `question-capture.sh` rather than hard-coding it to seed-agent's own plans.]*

The state machine is friction, not a wall. An operator in [gmode](06_9-gmode.html) can flip `plan_state` directly with `jq`. The approval gates depend on VERIFY actually firing the AskUserQuestion at the right stage; the `.yaml` injection assumes the phasic system fires on every phase entry; the validator can be bypassed by anyone willing to call the script directly. The architecture is the ceremony; the discipline rests on the agent honoring it, not on the gate being unbreakable. *[ref: validator-bypassable-architecture-is-ceremony | .claude/plugins/job_core/hooks/question-capture.sh:67-69 | The validator only fires when the question being asked starts with one of `[JOB-COMPLETE|PLAN-APPROVAL|YAML-APPROVAL]`. An operator who edits `data.json` directly, or calls `plan.sh approve-md` with HOOK_MODE=true set in their own shell, sidesteps the question-capture path entirely. The gate is a habit-shaping ceremony for the agent, not a tamper-proof seal against the operator. The plan-state machine is intentionally honest about its threat model: the gate exists to keep the agent disciplined; the operator has root.]*

## What the machine teaches

The plan-state machine is the final mechanism of the markov phasic brain. The cycle from [Essay 6.2](06_2-discipline-and-map.html) gave the brain phases. CONDENSE from [Essay 6.7](06_7-condense.html) gave it metabolism. The multiplier from [Essay 6.8](06_8-inverse-multiplier.html) gave it honest scope. Gmode from [Essay 6.9](06_9-gmode.html) gave it an off-cycle lane. The plan-state machine gives it a *horizon* — the capacity to hold the same plan across days, weeks, dozens of cycles, with the plan re-injecting itself at every phase boundary so the brain never forgets where the work has reached.

That is what makes the markov brain a brain, not a loop. A loop reacts. A brain remembers. The state machine is the substrate of that remembering, and the `.yaml` injection is the reflex that fires it back into the present every time the agent crosses a phase.

The cycle, the organ, the multiplier, gmode, the plan-state machine — each of these is a mechanism. None of them is the *thing they run inside*. Phases are plugins. CONDENSE is a plugin. Gmode rides on a plugin. The plan-state machine itself is split between `phase_plan` and `job_core`. Everything in this essay series is built on a standardized way of packaging cognitive mechanisms — the plugin kit. That is the subject of the next essay. *[ref: phases-are-plugins-state-machine-spans-two | .claude/plugins/ directory layout | The plugin directory shows the packaging: `phase_observe/`, `phase_plan/`, `phase_execute/`, `phase_verify/`, `phase_condense/` are each a plugin; `phasic_system/` is the orchestrator; `job_core/` owns the question-validator pipeline. The plan-state machine code lives across two of these — set-plan-file / approve-md / approve-yaml / seal-plan in `phase_plan/scripts/plan.sh`, and the `[PLAN-APPROVAL]` / `[YAML-APPROVAL]` validators in `job_core/hooks/question-capture.sh`. No single plugin owns the lifecycle; the kit is what lets two plugins coordinate one cognitive mechanism.]*

This is the architect's pivot point. Everything from [Essay 5.1's two-layer foundation](../b5/05_1-the-two-layer-foundation.html) through the markov phasic brain we have just finished mapping — was the seed agent's anatomy. You have seen the always-on cortex, the CLAUDE.md hierarchy, the phases, the cognitive metabolism organ, the multiplier dial, gmode's off-cycle lane, the plan-state machine's long-horizon memory. You have seen *how the seed agent thinks*. What you have not yet seen is how a new piece of that cognition gets *built*.

The phases and the waterfall are mechanisms. How do you BUILD a new phase that fits this design? How do you grow a new always-on plugin without breaking the substrate? How do you author your own marker, your own coaching voice, your own subagent definition, and have the rest of the architecture accept them as natively as the prototype's own?

[Essay 7 — The Plugin Kit](../b7/07_1-plugin-kit-foundation.html) is the answer. The brain is built. The kit is what lets you grow new pieces of it safely.
